# QUARTERMASTER — Implementation Guide

Binding conventions for every QUARTERMASTER service. Read alongside the [README](README.md) (architecture) and your service's `SPEC.md`. Wherever a service SPEC says "optional," it means runtime-configurable — implemented in full and dormant until configured — never skippable.

---

## 1. Identifiers, time, and data

- **IDs** — UUIDs, lowercase, string-serialized in every API. Resource IDs are generated by the owning service. A **user's identity** is their immutable directory principal (the Samba-AD `objectGUID`, carried as the OIDC subject); services key their records on that, never on a username or display name, which can change.
- **Timestamps** — UTC, ISO-8601 with a `Z` suffix and second precision on the wire; databases store `timestamptz`.
- **Timezone** — the campus is single-timezone. `PLATFORM_TZ` (e.g. `America/Los_Angeles`) is set identically on every host. All calendar semantics — "day", "week", "month", policy windows, and UI display — are computed in `PLATFORM_TZ`; APIs speak UTC and convert at the edges. Calendar periods are boundary-aligned (day = local midnight; week = local Monday 00:00; month = local 1st 00:00), never rolling.

## 2. HTTP and API conventions

- **Base path** — `/api/` for user and service APIs. Device APIs use a separate port (§6).
- **Content** — `application/json; charset=utf-8` both directions; snake_case field names.
- **Error envelope** — every non-2xx returns `{ "error": { "code": "...", "message": "...", "details": { ... } } }`. `code` is a stable machine string, `message` is human-readable English, `details` is optional structured context. HTTP status conveys the class: 400 validation, 401 unauthenticated, 403 denied, 404 not found, 409 conflict, 422 domain-rule violation, 429 rate-limited, 503 dependency unavailable.
- **Pagination** — cursor-based: request `?limit=&cursor=`, response `{ "items": [...], "next_cursor": "..." | null }`.
- **Idempotency** — mutating endpoints a client may retry accept an optional `Idempotency-Key` (a UUID). The server stores the first committed result under `(caller, key)` and replays it for retries, so a retry never applies the mutation twice. Authorization is re-evaluated on every attempt — a stored result is not a standing permission — and the key is retained for a fixed short window.
- **Health** — `GET /healthz` returns 200 with dependency checks (db, cache) or 503 with the failing checks.
- **Request correlation** — accept a caller-supplied `X-Request-Id` only when it is a single well-formed UUID; otherwise generate one. Echo it on the response, propagate it unchanged on causally-related outbound calls, and use it in logs and audit. It is untrusted correlation metadata and must never influence authentication, authorization, identity, or business logic.

## 3. Authentication and identity

- **One credential**, stored in Samba-AD. A service authenticates a browser user as an **OIDC client of Keycloak** (Authorization Code + PKCE); Keycloak federates the credential to Samba-AD. File access (TrueNAS SMB/NFS) authenticates via Kerberos/AD directly. No service stores passwords.
- On login a service receives the user's **identity** (directory subject + display name) and their **group memberships** as a token claim. It establishes its own session — an HTTP-only, `SameSite=Lax` cookie, with CSRF protection on mutations — and re-validates identity and claims per request.
- The **credential lifecycle** (email onboarding, self-service reset, MFA enrollment) belongs to Keycloak; password changes are written back to Samba-AD so web and file access stay on the same credential.
- **Dev/test login** — a hermetic stub-login (no external IdP, no directory) may be enabled by config in isolated dev/test environments only; production profiles must reject it. It is the sole auth bypass and is mutually exclusive with real identity settings (see §8, dev/prod parity).

## 4. Authorization

Authorization has two tiers.

- **Coarse (central, IT-managed)** — a small set of Keycloak group roles delivered in the token: `sysadmin`, `admin-<app>`, and department groups `group-*`. Services read these; IT manages nothing finer.
- **Fine (per app)** — each domain app owns its detailed permission model internally and may enforce it with a local policy engine (e.g. Cerbos). The coarse `admin-<app>` role bootstraps that app's internal admin authority, from which finer roles are assigned inside the app.

Every app implements these rules:
- **`sysadmin` = full access** within the app (IT troubleshooting), overriding all internal roles.
- **`admin-<app>`** unlocks the app's administrative surface.
- **Department gate** — an app may require membership in one or more `group-*` to be reachable at all. This gate is **enforced by the app and configured by the app's admin**; `sysadmin` and `admin-<app>` bypass it.
- A person is one privilege tier × any number of department groups (README §2).

## 5. Observability and audit

- Structured JSON logs to stdout, with at least `ts`, `level`, `service`, `request_id`, `msg`.
- An **append-only audit record** for every privileged mutation: `{ ts, actor (directory subject), action, object_type, object_id, before, after, request_id }`. Never updated or deleted.
- Prometheus metrics on a localhost `/metrics`.

## 6. Shared interlock device protocol

Equipment interlocks (dibs) and locked cabinets (shiny) use one identical node protocol. Nodes (ESP32 / Raspberry-Pi class) are policy-free: they energize only while the owning service says yes **and** the answer is still fresh.

- **Provisioning** — an admin registers a node in the owning service, links it to exactly one equipment item or storage unit, and receives a one-time per-node key (stored hashed). Firmware carries only the service URL, the node id, and the key.
- **Desired-state polling** — the node polls its service every few seconds for its desired state (`enabled`, a freshness TTL, and context such as an active session). It applies `enabled` only while the response is within the TTL.
- **Fail-safe** — each node has a provisioned fail state, `fail_enabled` or `fail_disabled`, applied locally when contact is lost past the TTL. Lock-type nodes default fail-secure; any node guarding an egress-relevant door must be provisioned fail-open (life-safety), confirmed at provisioning and audit-logged.
- **Safety invariant** — a running or energized session is **never** ended automatically: not at reservation end, not on a permission change, not during a service outage. The only mid-use stop is a deliberate manual action by an authorized operator. Cutting power to in-use equipment is a physical hazard, not an IT event.
- **Port security** — the device port is separate and **self-securing**: mandatory TLS (nodes pin the CA), the per-node key, and per-node rate limiting. It assumes an untrusted network shared with client devices; network placement is never treated as a security boundary.
- **Health** — nodes heartbeat periodically; the service marks a node offline after several missed heartbeats and alerts.

## 7. Testing standard

- **Test-driven (binding)** — the tests for a feature are written, and failing, **before** its code. Every bug fix begins with a regression test that reproduces the bug (and its near neighbours) and fails before the fix. Frontend UI is tested **comprehensively** — component rendering across states, user interactions (clicks, forms, validation, error/empty/loading), and the key user flows — not merely a boot check.
- **Coverage of security-sensitive paths is 100% (binding)** — authorization and access-control decisions, credential/token handling, and the interlock device port are covered to **100%, branches included**. The **steward services are security-sensitive in their entirety** — they mint `sysadmin`s and hold break-glass-equivalent tokens (README §2) — so each enforces a **100% coverage gate** as one of its `make` targets; every other service must still reach 100% on its own security-sensitive paths even where overall coverage is merely "comprehensive."
- **Hermetic** — tests depend only on localhost (Postgres/Redis via env-provided URLs), never on the network, real identity providers, organization-specific values, the host timezone/locale, or CPU architecture. Time is frozen or travelled for all calendar math (DST edges pinned to named instants). Tests are deterministic and pass in any order.
- **Make targets are the tests** — every gate is a local `make` target (`lint`, `test`, the service's own gates, `test-deploy`). CI (GitHub Actions) is a thin wrapper that only checks out, provides toolchains and service containers, and invokes those targets — it contains no test logic of its own. A CI failure that `make` cannot reproduce locally is a bug in the workflow.

## 8. Deployment

- **Deployable unit** — each service is a self-contained Docker Compose stack: its app containers plus its own Postgres and Redis on an internal network, built from its own repo, run as a non-root user. A host needs only Docker Engine + the Compose plugin. Physical topology (one host, one host per service, or a cluster) is a per-deployment choice and appears nowhere in code.
- **Configuration (binding)** — the **only configuration required to start any service is its `deploy/host.env` file** — no second config file, no bootstrap console step, nothing else on disk. Every deployment value comes from that single `.env`, its one home; images contain no organization-specific literal (domain, address, CIDR, timezone, identity settings, credentials), and the platform is organization-agnostic — such values flow only from the deployment profile into `host.env` into the container environment. All **runtime-tunable policy** (quotas, limits, permission models, mapping rules) is configured **in-app by admins**, never in a file, so `.env` stays limited to what a service needs to boot and connect.
- **Dev/prod parity (binding)** — dev and production run the **exact same path**: the same images, the same stack, the same deploy flow, the same code. The **only** difference is the `host.env` file — dev/test supplies local placeholders and the hermetic stub-login; production supplies the real Keycloak, directory, and TLS settings. There is no dev-only code path, image, or mode discriminator.
- **TLS edge** — an operator-provided reverse proxy terminates TLS for each service's web port and routes by hostname; the device port terminates its own TLS inside the stack (nodes pin the CA).

## 9. Shared UI conventions

- React + TypeScript + Vite + Tailwind + shadcn/ui; responsive and mobile-first; PWA-friendly. UI copy in English; times rendered in `PLATFORM_TZ` with an explicit zone label.
- **Shared design tokens** — `design-tokens.json` holds light and dark color palettes (identical semantic keys), a spacing scale, and a type scale, so every app feels like one platform. Every UI ships both themes, user-switchable, defaulting to the OS preference.
- **People directory** — every app provides a **People** page: a roster of its users, readable by any authenticated user, showing each person's SSO-guaranteed name and email and their app-relevant permission level. It exposes only what the directory already publishes (name, email). Admin configuration lives on a separate admin-only settings surface, not the roster.
- **Capability-aware controls** — a control the current user is not authorized to use is hidden or at least disabled/greyed, never a live button that only fails on click; the UI reflects the user's actual permissions. Runtime-configurable policy (quotas, limits, the permission model) is set by admins in-app, not baked into the build.
- **In-app help (binding)** — task-oriented, plain-language help, one topic per screen, linked from every friction point, so a denied action or a blocked flow is one click from its explanation and a user can recover without leaving the app.
